# Phase 2 — Academic Core: Grades, Classes, Subjects, Teaching Assignments & Weekly Schedule

**Developer:** Dev 2  
**Complexity:** High  
**Dependencies:** Phase 1 (User entity, AuthService, role identities, SubscriptionAccessMiddleware must exist)

---

## Objective

Deliver the complete academic structure vertical slice. Admins can define curriculum grades (e.g., Grade 1 through Grade 12 split by stage), create classes within grades, define subjects, assign teachers to subjects within classes, and publish weekly class schedules. Teachers can view their assigned classes and upcoming sessions. Students can view their enrolled class schedule. Every endpoint in this phase is protected by the subscription middleware from Phase 1.

---

## 1. Functional Requirements

- Admin creates and manages school grades (Ibtedaey 1–6, E'dady 1–3, Thanwy 1–3)
- Admin creates classes within a grade (e.g., "5A", "5B")
- Admin defines subjects per grade (Arabic, Math, English, Science, etc.)
- Admin assigns a teacher to a subject within a specific class (TeachingAssignment)
- Admin enrolls students in a class (StudentClass)
- Teacher can view all their teaching assignments
- Teacher can create sessions (scheduled live classes) for a class-subject
- Students enrolled in a class can view the weekly schedule of upcoming sessions
- Schedule conflicts must be detected: same class cannot have two sessions at overlapping times
- Sessions reference a third-party embed URL (Zoom/Teams link provided by teacher) — not generated by the system

```csharp
public record TimeSlot(DateTime Start, int DurationMinutes)
{
    public DateTime End => Start.AddMinutes(DurationMinutes);
    public bool OverlapsWith(TimeSlot other) =>
        Start < other.End && End > other.Start;
}
```

### Domain Services

```csharp
public class ScheduleConflictChecker
{
    // Checks if a proposed session overlaps with any existing session for the same class
    public bool HasConflict(IEnumerable<Session> existingSessions, TimeSlot proposed)
    {
        return existingSessions
            .Where(s => s.Status != SessionStatus.Cancelled)
            .Any(s => new TimeSlot(s.ScheduledAt, s.DurationMinutes).OverlapsWith(proposed));
    }
}
```

### Command Handler Logic Notes

**`ScheduleSessionHandler`**
1. Load `TeachingAssignment` by id — verify `TeacherUserId` matches calling teacher
2. Load existing sessions for the class on that date
3. Use `ScheduleConflictChecker.HasConflict` — return `409 Conflict` with message if overlap found
4. Create `Session` via factory, persist
5. Publish `SessionScheduledEvent` → consumed by Notifications service (Phase 6)

**`EnrollStudentHandler`**
1. Verify student user exists and has Student role
2. Verify class exists and is active
3. Check `MaxCapacity` against current enrollment count — return `409` if full
4. Check student not already enrolled in another class for same grade/year
5. Create `StudentClass`, persist

### Caching Strategy

- Grades list: cached in Redis for 1 hour (rarely changes), key: `grades:all`
- Subjects by grade: cached 1 hour, key: `subjects:grade:{gradeId}`
- Invalidated on any admin create/update

### Weekly Calendar Component

The `weekly-calendar` component is the primary UI for students. It renders a 5-column (Mon–Fri) grid. Each session is placed at the correct time slot based on `scheduledAt`. ## 9. Risks

| Risk | Mitigation |
|---|---|
| Teacher provides invalid embed URL | Validate URL format in command validator; warn but do not block (URL may not be created yet) |
| Timezone differences for Egyptian students abroad | Store all `ScheduledAt` as UTC; frontend converts to user's local timezone via `Intl.DateTimeFormat` |
| Class capacity edge case under concurrent enrollment | Use optimistic concurrency on Class enrollment count or serialize via Redis lock |
